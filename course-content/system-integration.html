Overview

Welcome to System Integration! This week, you'll bring together all the skills you've learned—mechanics, electronics, programming, and interfaces—into a complete, working system. This is where your final project comes together.

Throughout the course, you've learned individual skills: CAD design, electronics, programming, networking, interfaces. Now it's time to integrate everything into a cohesive system that solves a real problem. We'll focus on mapping all subsystems, planning integration steps, identifying risks, and testing the complete system.

This week is the culmination of your UNIPOD journey. By the end, you'll have a working prototype that demonstrates how all components work together to solve a real problem in your community. You'll also document your system clearly so others can understand, reproduce, and build upon your work.

Learning Outcomes

By the end of this week, you will be able to:

  • Map all subsystems of your project including power, sensors, outputs, structure, enclosure, and interface
  • Create system diagrams showing how all components connect and communicate
  • Plan integration steps and identify dependencies between subsystems
  • Identify risks and potential failure points in your system
  • Test the complete system end-to-end and verify all components work together
  • Iterate and improve based on testing results and peer feedback
  • Document the system comprehensively so others can understand and reproduce it

Core Concepts & Theory

1. System Architecture

A complete system consists of multiple subsystems that must work together:

  • Power Subsystem:
    • Power source (battery, wall adapter, solar)
    • Voltage regulation (convert to required voltages)
    • Power distribution (route power to all components)
    • Power management (on/off switches, sleep modes)
  • Sensor Subsystem:
    • Input devices (temperature, light, distance, etc.)
    • Signal conditioning (amplifiers, filters)
    • Data acquisition (ADC, digital inputs)
  • Processing Subsystem:
    • Microcontroller(s)
    • Communication between processors (if multiple)
    • Data processing and decision-making logic
  • Output Subsystem:
    • Actuators (motors, relays, displays)
    • Driver circuits (transistors, motor drivers)
    • Feedback mechanisms
  • Communication Subsystem:
    • Wired protocols (I2C, SPI, UART)
    • Wireless protocols (Wi-Fi, Bluetooth, LoRa)
    • Network topology (star, mesh, bus)
  • Interface Subsystem:
    • User interface (buttons, displays, web interface)
    • Data visualization
    • Control mechanisms
  • Mechanical Subsystem:
    • Structure and enclosure
    • Mounting and fastening
    • Protection (weather, dust, impact)

2. Integration Planning

Successful integration requires careful planning:

  • Identify Dependencies:
    • What must work before other things can work?
    • Example: Power must work before sensors can be read
    • Example: Communication must work before distributed sensors can send data
  • Integration Order:
    • Start with core functionality (power + processing)
    • Add inputs (sensors)
    • Add outputs (actuators)
    • Add communication
    • Add interface
    • Add enclosure and final assembly
  • Testing Strategy:
    • Test each subsystem independently first
    • Test subsystems in pairs (e.g., sensor + processing)
    • Test complete system end-to-end
    • Test under realistic conditions

3. Risk Identification and Mitigation

Identify potential problems before they occur:

  • Power Risks:
    • Insufficient power supply capacity
    • Voltage drops under load
    • Battery drain too fast
    • Mitigation: Calculate power requirements, test with full load, use appropriate power supplies
  • Communication Risks:
    • Data loss or corruption
    • Timing issues (synchronization)
    • Network failures
    • Mitigation: Add error checking, timeouts, retry logic, fallback modes
  • Mechanical Risks:
    • Components not fitting together
    • Vibration or impact damage
    • Weather exposure
    • Mitigation: Test fits early, design for robustness, use appropriate enclosures
  • Software Risks:
    • Bugs causing system crashes
    • Memory leaks
    • Race conditions
    • Mitigation: Test thoroughly, add error handling, use watchdog timers

4. System Documentation

Comprehensive documentation is essential:

  • System Block Diagram: High-level overview showing all subsystems and connections
  • Wiring Diagrams: Detailed connections between all components
  • Software Architecture: Code structure, data flow, communication protocols
  • Power Budget: Power consumption of each component, total power requirements
  • Bill of Materials (BOM): Complete list of components with part numbers and sources
  • Assembly Instructions: Step-by-step guide to building the system
  • Operating Instructions: How to use the system, troubleshooting guide

Mini-Lab: Create a System Diagram for Your Project

Let's create a comprehensive system diagram that maps all components and their connections—essential for integration planning.

Step 1: List All Components

Create a list of every component in your system:

  • Microcontrollers/processors
  • Sensors (temperature, light, distance, etc.)
  • Outputs (motors, LEDs, displays, relays)
  • Power supplies and regulators
  • Communication modules (Wi-Fi, Bluetooth, etc.)
  • Mechanical parts (enclosure, mounts, etc.)

Step 2: Identify Subsystems

Group components into subsystems:

  • Power subsystem: battery, regulator, distribution
  • Sensor subsystem: all input devices
  • Processing subsystem: microcontrollers
  • Output subsystem: all actuators
  • Communication subsystem: networking components
  • Interface subsystem: displays, buttons, web interface

Step 3: Map Connections

For each connection, identify:

  • What connects to what (component A → component B)
  • Type of connection (power, data, control signal)
  • Protocol used (I2C, SPI, UART, analog, digital)
  • Physical connection (wires, PCB traces, wireless)

Step 4: Create the Diagram

Use drawing software (Draw.io, Inkscape, or paper):

  1. Draw boxes for each subsystem
  2. Draw boxes for major components within subsystems
  3. Draw arrows showing connections (label with protocol/type)
  4. Use different colors for different connection types (power=red, data=blue, etc.)
  5. Add labels and annotations

Step 5: Add Power Flow

Show power distribution:

  • Power source → regulators → components
  • Label voltages (5V, 3.3V, 12V, etc.)
  • Show current paths

Step 6: Add Data Flow

Show information flow:

  • Sensors → processing → outputs
  • Communication paths
  • Interface connections

Assignments & Documentation Requirements

Individual Assignment

Produce a system diagram for your final project and show that all components are connected and working together:

  • Create a comprehensive system block diagram showing:
    • All subsystems (power, sensors, processing, outputs, communication, interface, mechanical)
    • All major components within each subsystem
    • All connections between components (power, data, control signals)
    • Communication protocols used (I2C, SPI, UART, Wi-Fi, etc.)
  • Demonstrate that all components work together:
    • Video showing complete system running through its main use case
    • Photos showing all components connected and operating
    • Documentation of integration steps and any issues encountered
  • Create a power budget:
    • List power consumption of each component
    • Calculate total power requirements
    • Verify power supply is adequate

Group Assignment (Optional but Recommended)

As a UNIPOD team, conduct informal "integration reviews":

  • Review each other's system diagrams
  • Help identify:
    • Missing components or connections
    • Potential failure points
    • Power issues
    • Communication problems
    • Mechanical issues
  • Provide constructive feedback and suggestions
  • Document common issues and solutions for future reference

Documentation Checklist

Item Description
System Block Diagram High-level diagram showing all subsystems, components, and connections. Use clear labels and colors.
Wiring Diagrams Detailed diagrams showing all electrical connections, pin assignments, and component values.
Power Budget Table showing power consumption of each component, total power requirements, and power supply specifications.
Software Architecture Documentation of code structure, data flow, communication protocols, and software components.
Integration Steps Step-by-step guide showing how you integrated subsystems, in what order, and why.
Testing Documentation Records of tests performed, results, issues found, and how they were resolved.
Complete System Demo Video showing the complete system running through its main use case, demonstrating all functionality.
Bill of Materials (BOM) Complete list of all components with part numbers, quantities, sources, and costs.
Open Issues and Future Improvements List of known issues, limitations, and ideas for future improvements beyond the course.
Tip: Your system diagram should be clear enough that someone else could understand your project and potentially reproduce it. Include enough detail but keep it readable.

Tips for Success

  • Start with a plan: Create system diagram before building—it helps identify issues early.
  • Test incrementally: Test each subsystem independently before integrating.
  • Document as you go: Don't wait until the end—document integration steps as you do them.
  • Expect problems: Integration always reveals issues. Plan time for debugging and iteration.
  • Get peer feedback: Others can spot issues you might miss.
  • Test under realistic conditions: Test in the environment where the system will actually be used.
  • Plan for failure: Add error handling, fallback modes, and graceful degradation.
  • Keep it simple: Don't overcomplicate—simple systems are more reliable and easier to debug.
