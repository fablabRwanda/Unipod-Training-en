Overview

Welcome to Embedded Programming! This week, you'll learn to write code for microcontrollers—the tiny computers that power everything from sensors to robots. Whether you're programming a board you designed last week or using a development board, you'll learn to read inputs, control outputs, and create interactive behaviors.

In many African contexts, being able to program custom hardware opens doors to creating solutions for local challenges—from agricultural monitoring systems to water quality sensors. We'll explore multiple programming approaches: Arduino (beginner-friendly), PlatformIO (more powerful), and plain C (maximum control). By the end of the week, you'll understand how to set up toolchains, write and debug code, and upload programs to your hardware.

This week builds directly on Electronics Design and Electronics Production (where you designed and fabricated your board) and prepares you for Input Devices and Output Devices (where you'll connect sensors and actuators). By the end, you'll have a working microcontroller board that responds to inputs and controls outputs—the foundation of any interactive project.

Core Concepts & Theory

1. Understanding Microcontroller Architecture

A microcontroller is a complete computer on a single chip. Unlike your laptop, it's designed for real-time control of physical systems. Key components:

  • CPU (Central Processing Unit): Executes your program instructions. Most UNIPOD boards use AVR (ATtiny, ATmega) or ARM (ESP32, STM32) processors running at 8-16 MHz.
  • Memory:
    • Flash: Stores your program code (non-volatile, persists when power is off)
    • SRAM: Stores variables while program runs (volatile, cleared on power loss)
    • EEPROM: Stores data that persists (like settings)
  • GPIO Pins: General Purpose Input/Output pins that can read sensors or control outputs. Can be configured as digital (HIGH/LOW) or analog (ADC for reading, PWM for controlling brightness/speed).
  • Clock: Determines how fast the CPU runs. External crystals provide accurate timing; internal oscillators are simpler but less precise.
  • Peripherals: Built-in hardware like timers, UART (serial communication), SPI, I2C buses.

2. Digital vs. Analog I/O

Understanding the difference between digital and analog is crucial:

  • Digital I/O: Pins can be HIGH (usually 5V or 3.3V) or LOW (0V). Perfect for buttons, LEDs, switches. Use digitalRead() and digitalWrite() in Arduino, or direct port manipulation in C.
  • Analog Input (ADC): Reads voltage levels between 0V and VCC (e.g., 0-1023 on 10-bit ADC). Used for sensors like potentiometers, light sensors, temperature sensors. Use analogRead() in Arduino.
  • Analog Output (PWM): Simulates analog output by rapidly switching between HIGH and LOW. Duty cycle (percentage of time HIGH) controls brightness of LEDs or speed of motors. Use analogWrite() in Arduino.

3. Programming Approaches

You have several options for programming microcontrollers:

  • Arduino IDE (Recommended for beginners):
    • Pros: Simple, lots of libraries, huge community, works on any OS
    • Cons: Less control over low-level details, can be slower
    • Best for: Rapid prototyping, learning, projects that don't need precise timing
  • PlatformIO (Recommended for intermediate users):
    • Pros: More powerful, better code organization, supports multiple platforms
    • Cons: Steeper learning curve, requires more setup
    • Best for: Larger projects, professional development, when you need more control
  • Plain C with AVR-GCC (Advanced):
    • Pros: Maximum control, smallest code size, precise timing
    • Cons: Complex, requires understanding of hardware registers, more error-prone
    • Best for: When you need every bit of performance or memory

4. Programming Interfaces

To upload code to your microcontroller, you need a programmer:

  • ISP (In-System Programmer): 6-pin header that connects directly to microcontroller. Works with any AVR chip. You can use an Arduino as an ISP, or dedicated programmers like USBasp.
  • USB-to-Serial: Many boards (like Arduino Uno) have built-in USB-to-serial chips (FTDI, CH340, CP2102). These allow direct programming via USB cable.
  • Bootloader: Some boards come with bootloaders that allow programming without a separate programmer. Convenient but uses flash memory.

Mini-Lab: Blink an LED and Read a Button

Let's start with the classic "Hello World" of embedded programming: blinking an LED and reading a button.

Step 1: Set Up Your Development Environment

  1. Install Arduino IDE:
    • Download from arduino.cc
    • Install drivers for your programmer (USBasp, CH340, etc.)
    • Open Arduino IDE and go to Tools → Board → Select your board (e.g., "ATtiny24/44/84" or "Arduino Uno")
    • Select the correct programmer under Tools → Programmer
  2. Connect your board:
    • If using ISP: Connect 6 wires (VCC, GND, MISO, MOSI, SCK, RST) between programmer and board
    • If using USB: Simply connect USB cable

Step 2: Write the Blink Program

Open a new sketch in Arduino IDE and write:

// Pin definitions
#define LED_PIN 0      // Change to match your board's LED pin
#define BUTTON_PIN 1  // Change to match your button pin

void setup() {
  // Initialize serial communication (for debugging)
  Serial.begin(9600);
  
  // Configure LED pin as output
  pinMode(LED_PIN, OUTPUT);
  
  // Configure button pin as input with pull-up resistor
  pinMode(BUTTON_PIN, INPUT_PULLUP);
  
  Serial.println("Program started!");
}

void loop() {
  // Read button state (LOW when pressed due to pull-up)
  int buttonState = digitalRead(BUTTON_PIN);
  
  if (buttonState == LOW) {
    // Button is pressed - turn LED on
    digitalWrite(LED_PIN, HIGH);
    Serial.println("Button pressed - LED ON");
  } else {
    // Button not pressed - turn LED off
    digitalWrite(LED_PIN, LOW);
    Serial.println("Button released - LED OFF");
  }
  
  // Small delay to avoid reading too fast
  delay(50);
}

Step 3: Upload and Test

  1. Click "Verify" (checkmark icon) to compile your code. Fix any errors.
  2. Click "Upload" (arrow icon) to program your board.
  3. Open Serial Monitor (Tools → Serial Monitor) to see debug messages.
  4. Press the button and watch the LED turn on/off!

Step 4: Experiment

  • Change the delay value - what happens?
  • Add a second LED and make them blink alternately
  • Use analogRead() to read a potentiometer and control LED brightness with analogWrite()

Recommended Software Tools

Arduino IDE (Recommended for Beginners)
  • Free and simple: Perfect for learning embedded programming
  • Cross-platform: Works on Windows, Mac, Linux
  • Huge library ecosystem: Thousands of libraries for sensors, displays, communication
  • Built-in serial monitor: Easy debugging
  • Download: arduino.cc
PlatformIO (For Advanced Users)
  • More powerful: Better code organization, dependency management
  • Multi-platform: Supports Arduino, ESP32, STM32, and more
  • Professional features: Unit testing, code linting, version control integration
  • Install: Extension for VS Code or standalone IDE
  • Website: platformio.org
Recommendation for UNIPOD labs: Start with Arduino IDE. It's free, simple, well-documented, and works on any computer. Once you're comfortable, you can explore PlatformIO for more advanced projects.

Tips for Success

  • Start simple: Don't try to write complex code on your first attempt. Begin with blink, then add one feature at a time.
  • Use Serial.print() liberally: Print variable values, program state, and debug messages. It's your best friend for finding bugs.
  • Check your pin numbers: Make sure your code uses the correct pin numbers for your board. ATtiny pin numbers are different from Arduino Uno!
  • Verify connections: Before blaming your code, double-check that wires are connected correctly. Use a multimeter if needed.
  • Read datasheets: When in doubt, check the microcontroller datasheet for pin functions, voltage levels, and capabilities.
  • Test incrementally: Don't write all your code at once. Test each function separately, then combine them.
  • Save working versions: Before making big changes, save a copy of working code. You can always go back!
  • Ask for help: If you're stuck, ask your lab manager or classmates. Programming has many "gotchas" that experienced developers know about.
