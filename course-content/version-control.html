Core Concepts & Theory

1. What is Version Control?

Version control is a system that tracks changes to files over time. Think of it as a time machine for your work:

  • Saves history: Every change is recorded, so you can go back to any previous version
  • Enables collaboration: Multiple people can work on the same project without overwriting each other's work
  • Prevents loss: If you delete something by mistake, you can recover it
  • Shows progress: You can see how your project evolved over time
  • Enables experimentation: Try new ideas in branches without breaking your main work

2. Git Basics

Git is the most popular version control system. Key concepts:

  • Repository (repo): A folder that Git tracks. Contains your project files and Git history.
  • Commit: A snapshot of your files at a specific point in time. Like saving a checkpoint.
  • Branch: A parallel version of your project. Lets you work on features without affecting the main code.
  • Merge: Combining changes from one branch into another.
  • Remote: A copy of your repository stored online (GitHub, GitLab, etc.).
  • Push: Uploading your commits to a remote repository.
  • Pull: Downloading commits from a remote repository.

3. Git Workflow

Typical workflow for documenting your work:

  1. Make changes: Edit files, add photos, write documentation
  2. Stage changes: git add . (tell Git which files to include)
  3. Commit: git commit -m "Description" (save a snapshot with a message)
  4. Push: git push (upload to remote repository)

This workflow lets you save your work frequently and share it with others.

4. Documentation Best Practices

Good documentation helps others (and future you) understand your work:

  • Be clear and organized: Use headings, lists, and structure
  • Include visuals: Photos, diagrams, and screenshots are worth a thousand words
  • Explain your process: Not just what you did, but why and how
  • Document failures: What didn't work and how you fixed it is valuable learning
  • Include files: Link to or embed design files, code, and resources
  • Update regularly: Document as you work, not just at the end
  • Make it reproducible: Include enough detail that others could recreate your work

5. Documentation Platforms

Choose a platform that works for your context:

  • GitHub Pages:
    • Free hosting for static websites
    • Integrated with Git—push code, website updates automatically
    • Use: HTML/CSS/JavaScript sites, Jekyll, or MkDocs
    • Best for: When you already use GitHub
  • MkDocs:
    • Documentation generator from Markdown files
    • Simple, clean, professional look
    • Use: When you want a documentation-focused site
    • Best for: Technical documentation
  • Plain HTML:
    • Simple, works everywhere, no dependencies
    • Use: When you want maximum control and simplicity
    • Best for: Simple portfolios, offline capability
  • GitLab Pages:
    • Similar to GitHub Pages but on GitLab
    • Use: If you prefer GitLab over GitHub

Mini-Lab: Set Up Git and Create Your First Repository

Let's set up Git and create your first repository for documenting your UNIPOD journey.

Step 1: Install Git

  1. Windows:
    • Download from git-scm.com
    • Run installer, use default settings
    • Git Bash will be installed (command-line tool)
  2. Mac:
    • Install Xcode Command Line Tools: xcode-select --install
    • Or download from git-scm.com
  3. Linux:
    • Ubuntu/Debian: sudo apt install git
    • Fedora: sudo dnf install git

Step 2: Configure Git

Set your name and email (used in commits):

git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

Step 3: Create a GitHub Account

  1. Go to github.com
  2. Sign up for a free account
  3. Verify your email

Step 4: Create Your First Repository

  1. On GitHub, click "New repository"
  2. Name it: unipod-portfolio (or similar)
  3. Make it public (so others can see your work)
  4. Initialize with README (optional)
  5. Click "Create repository"

Step 5: Clone Repository to Your Computer

# Copy the repository URL from GitHub (click green "Code" button)
git clone https://github.com/yourusername/unipod-portfolio.git
cd unipod-portfolio

Step 6: Create Your First Documentation Page

Create a simple HTML file:

<!DOCTYPE html>
<html>
<head>
    <title>My UNIPOD Journey</title>
</head>
<body>
    <h1>Welcome to My UNIPOD Portfolio</h1>
    <h2>About Me</h2>
    <p>Your name, background, and what you want to make.</p>
    <h2>Weekly Assignments</h2>
    <p>Links to your weekly documentation will go here.</p>
</body>
</html>

Save as index.html

Step 7: Commit and Push

# Stage your files
git add index.html

# Commit with a message
git commit -m "Add initial portfolio page"

# Push to GitHub
git push origin main

Step 8: Enable GitHub Pages

  1. On GitHub, go to Settings → Pages
  2. Select "main" branch as source
  3. Click "Save"
  4. Your site will be available at: https://yourusername.github.io/unipod-portfolio

Tips for Success

  • Commit frequently: Commit after each significant change, not just at the end of the day.
  • Write clear commit messages: Describe what changed and why. Future you will thank present you!
  • Push regularly: Push to GitHub frequently to backup your work and share progress.
  • Use branches for experiments: Try new things in branches without breaking your main work.
  • Document as you go: Don't wait until the end—document while you work so you don't forget details.
  • Include photos: Photos are essential for documentation. Take many photos as you work.
  • Organize files clearly: Use folders to organize by week or topic. Clear organization helps you and others.
  • Test your website: Make sure your published site works and looks good before sharing.
  • Keep it simple: Start with simple HTML. You can add complexity later if needed.
  • Ask for help: Git can be confusing at first. Ask your lab manager or classmates for help.
